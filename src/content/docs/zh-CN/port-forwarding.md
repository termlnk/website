# 端口转发

Termlnk 提供独立的端口转发面板，用 SSH 连接创建、启动和停止转发规则。规则与 SSH 主机配置复用同一套主机树、认证方式、ProxyJump 链和主机密钥验证。

![端口转发规则列表](/images/docs/termlnk-port-forwarding-overview.png)

## 三种转发类型

| 类型 | SSH 等价参数 | 监听位置 | 目标位置 | 典型用途 |
| --- | --- | --- | --- | --- |
| 本地转发（Local） | `-L` | 本机 | SSH 中转主机能访问的目标服务 | 在本机访问远程内网数据库、管理后台或 HTTP 服务 |
| 远程转发（Remote） | `-R` | 远程 SSH 服务器 | 本机能访问的目标服务 | 把本机开发服务临时暴露给远程环境 |
| 动态转发（Dynamic） | `-D` | 本机 SOCKS5 端口 | 由客户端请求决定 | 通过 SSH 建立临时 SOCKS5 代理 |

## 创建规则

打开侧边栏的端口转发入口，点击「+」后选择 Local、Remote 或 Dynamic。规则表单包含类型切换、拓扑图和连接字段。

![Local 转发表单](/images/docs/termlnk-port-forwarding-local-form.png)

通用字段：

- **标签** — 用于在规则列表中识别规则。
- **监听地址** — tunnel listener 绑定的地址，例如 `127.0.0.1` 或 `0.0.0.0`。
- **监听端口** — listener 绑定的端口。端口可以填 `0`，由系统或远端 sshd 自动分配可用端口，实际端口会体现在运行状态里。
- **主机** — 从 SSH 主机树中选择，Termlnk 会复用该主机的认证、ProxyJump、SOCKS5 代理和主机密钥配置。
- **目标地址 / 目标端口** — Local 和 Remote 需要填写，Dynamic 不需要。

## 本地端口转发（Local）

本地端口转发等价于 `ssh -L`。Termlnk 在本机启动一个 TCP listener，客户端连接这个本地端口后，流量会通过 SSH 主机转发到目标服务。

```text
本机客户端 -> 本机监听端口 -> SSH 主机 -> 目标服务
```

### 适合场景

- 在本机浏览器访问远程内网 Web 管理台。
- 用本机数据库客户端连接远程内网数据库。
- 调试只能从跳板机访问的 HTTP、Redis、MySQL、PostgreSQL 等服务。

### 字段含义

| 字段 | 含义 | 示例 |
| --- | --- | --- |
| 监听地址 | 本机监听地址。只给本机使用时推荐 `127.0.0.1` | `127.0.0.1` |
| 本地端口 | 本机开放给客户端访问的端口 | `8009` |
| 中转主机 | 负责建立 SSH tunnel 的主机 | `prod-relay` |
| 目标地址 | 从中转主机视角可以访问的地址 | `example.com` 或 `10.0.0.12` |
| 目标端口 | 目标服务端口 | `80`、`3306`、`5432` |

### 示例

```text
监听地址: 127.0.0.1
本地端口: 8009
中转主机: prod-relay
目标地址: example.com
目标端口: 80
```

启动后，在本机访问：

```text
http://127.0.0.1:8009
```

请求路径是：

```text
127.0.0.1:8009 -> prod-relay -> example.com:80
```

### 注意事项

- 如果端口被占用，规则会进入失败状态。换一个本地端口，或把端口填 `0` 让系统分配。
- 目标地址不是从本机解析，而是从 SSH 主机侧访问；如果 SSH 主机无法访问目标，转发会建立成功但连接会失败。
- 把监听地址设为 `0.0.0.0` 会允许局域网内其他机器连接你的本机端口，只在明确需要共享时使用。

## 远程端口转发（Remote）

远程端口转发等价于 `ssh -R`。Termlnk 请求远程 SSH 服务器监听一个端口；远端有客户端连接该端口时，流量会通过 SSH tunnel 回到本机，再连接本机可访问的目标服务。

```text
远端客户端 -> 远程 SSH 服务器监听端口 -> SSH tunnel -> 本机 -> 目标服务
```

### 适合场景

- 把本机开发服务临时暴露给远程服务器或远端网络。
- 在远程服务器上回调本机运行的 webhook、调试服务或 API mock。
- 让远端环境访问只在本机网络中可见的服务。

### 字段含义

| 字段 | 含义 | 示例 |
| --- | --- | --- |
| 远程主机 | 负责监听远端端口的 SSH 服务器 | `prod-relay` |
| 监听地址 | 远程 SSH 服务器上的监听地址 | `127.0.0.1` 或 `0.0.0.0` |
| 远程端口 | 远程 SSH 服务器上开放的端口 | `9000` |
| 目标地址 | 从本机视角可以访问的地址 | `localhost`、`127.0.0.1` |
| 目标端口 | 本机或本机网络中的目标服务端口 | `3000`、`9001` |

### 示例

```text
远程主机: prod-relay
监听地址: 127.0.0.1
远程端口: 9000
目标地址: localhost
目标端口: 3000
```

启动后，在远程 SSH 服务器上访问：

```text
curl http://127.0.0.1:9000
```

请求路径是：

```text
prod-relay:9000 -> SSH tunnel -> 本机 localhost:3000
```

### 允许外部访问

如果希望远程服务器以外的机器也能访问该远程端口，需要同时满足：

- 规则的监听地址填写 `0.0.0.0`。
- 远程服务器的 `sshd_config` 允许远程转发绑定外部地址，通常需要 `GatewayPorts yes` 或 `GatewayPorts clientspecified`。
- 远程服务器防火墙、安全组或云厂商网络策略放行该端口。

### 注意事项

- 远程端口由 sshd 监听，所以端口冲突发生在远程服务器上，而不是本机。
- 目标地址从本机视角解析；`localhost:3000` 指的是运行 Termlnk 的这台机器。
- 把监听地址设为 `0.0.0.0` 会扩大暴露面，建议只给临时调试使用，并及时停止规则。

## 动态端口转发（Dynamic）

动态端口转发等价于 `ssh -D`。Termlnk 在本机启动一个 SOCKS5 代理，应用程序把请求交给这个代理后，代理会根据每个请求的目标地址，通过 SSH 主机建立对应连接。

```text
本机应用 -> 本机 SOCKS5 代理 -> SSH 主机 -> 请求指定的目标地址
```

### 适合场景

- 临时让浏览器通过 SSH 主机访问远端网络。
- 让包管理器、API 调试工具或命令行程序走 SSH 代理。
- 不想为每个目标服务单独创建 Local 规则时，使用一个 SOCKS5 入口处理多个目标。

### 字段含义

| 字段 | 含义 | 示例 |
| --- | --- | --- |
| 监听地址 | 本机 SOCKS5 代理监听地址 | `127.0.0.1` |
| 本地端口 | SOCKS5 代理端口 | `8981` |
| 中转主机 | 负责向目标地址发起连接的 SSH 主机 | `prod-relay` |

Dynamic 不填写目标地址和目标端口，因为目标由每个 SOCKS5 请求决定。当前实现支持 SOCKS5 `CONNECT` 请求，目标可以是 IPv4、域名或 IPv6。

### 示例

```text
监听地址: 127.0.0.1
本地端口: 8981
中转主机: prod-relay
```

启动后得到代理：

```text
socks5://127.0.0.1:8981
```

在命令行中使用：

```bash
curl --socks5-hostname 127.0.0.1:8981 https://example.com
```

### 注意事项

- 使用 `socks5h` 或 `--socks5-hostname` 可以让域名解析也通过代理侧完成，避免本地 DNS 泄漏或解析不到内网域名。
- Dynamic 不是 HTTP 代理，应用需要支持 SOCKS5。
- 如果监听地址设为 `0.0.0.0`，其他机器也可能使用这个代理，请确认网络边界和访问控制。

## 启动和停止

规则创建后不会永久自动运行。列表中的播放按钮用于启动规则，停止按钮用于关闭正在运行的 tunnel。

运行时状态包括：

- `Idle` / `Starting` / `Authenticating` / `Active` / `Failed` / `Stopping` / `Closed`
- 当前活跃连接数
- 总连接数
- 入站 / 出站总流量
- 入站 / 出站实时速率

如果启动过程中需要交互式认证、修改密码或确认主机密钥，Termlnk 会弹出对应的认证对话框。首次连接仍遵循 TOFU（Trust On First Use）主机密钥策略。

## 与 SSH 配置的关系

端口转发规则通过 `hostId` 引用 SSH 主机，而不是单独保存一份 SSH 登录信息。因此：

- 修改主机的密码、私钥、SSH Agent、ProxyJump 或 SOCKS5 代理配置后，转发规则会复用最新配置。
- 删除或改动主机可能影响引用它的转发规则。
- 同一主机上的多条规则可以共享底层 SSH socket，减少重复连接开销。

## 常见示例

### 本地访问远程内网服务

```text
127.0.0.1:8009 -> www.termlnk.com:80
```

浏览器访问 `http://127.0.0.1:8009`，流量通过 SSH 中转主机到达目标 HTTP 服务。

### 远程访问本机开发服务

```text
0.0.0.0:9000 -> localhost:9001
```

远程 SSH 服务器监听 `9000`，访问该端口的流量会回到本机的 `localhost:9001`。

### 建立 SOCKS5 代理

```text
socks5://127.0.0.1:8981
```

把浏览器、包管理器或调试工具的代理设置为该地址，即可让流量通过 SSH 主机转发。
